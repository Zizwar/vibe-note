# تقرير شامل: تطبيق Vibe Note — التحليل وخارطة طريق الإصدار الثاني (v2)

> وثيقة تحليلية معمّقة تغطّي وصف التطبيق، ميزات الإصدار الأول، البنية التقنية، نقاط القوة والضعف، ومقترحات تطوير الإصدار الثاني مع خارطة طريق تنفيذية.
>
> **تاريخ التقرير:** 9 يونيو 2026 · **الفرع:** v1 · **الإصدار الحالي:** 0.9.1 (app.json: 1.0.0)

---

## 1. نظرة عامة ووصف التطبيق

**Vibe Note** (الاسم في المستودع: `proomy-note`) هو تطبيق جوّال **لإدارة البرومبتات (Prompts)** موجّه للأشخاص الذين يستخدمون أدوات الذكاء الاصطناعي بشكل يومي. الفكرة الأساسية أنّ البرومبتات الجيدة تضيع عادةً بين تبويبات المتصفح وتطبيقات الملاحظات وسجلّات المحادثات، فجاء التطبيق ليمنحها "بيتاً منظّماً".

يتيح التطبيق:
- **حفظ البرومبتات** مع عنوان، تصنيف، وسم منصّة الذكاء الاصطناعي، ووسوم مخصّصة (tags).
- **القوالب المتغيّرة (Variable Templates):** كتابة متغيّرات داخل النص مثل `{{topic}}` أو `{{tone:formal|casual|professional}}`، فيحوّلها التطبيق تلقائياً إلى نموذج تفاعلي يملؤه المستخدم.
- **بحث فوري** عبر العناوين والمحتوى والوسوم باستخدام بحث نصّي كامل (FTS5).
- **مساعد ذكاء اصطناعي** لاستيراد نصوص خام وتنظيمها تلقائياً، وتوليد برومبتات جديدة من وصف، وتحسين البرومبتات القائمة، وتحليل جودتها، إضافة إلى محادثة حرّة.
- **تتبّع الاستخدام** (عدد مرّات الاستخدام، آخر استخدام، والقيم الأخيرة المُدخلة في المتغيّرات).
- **التصدير والاستيراد** للمكتبة بالكامل أو لبرومبت واحد عبر ملفات بصيغة `.vibe`.

**الفلسفة المميّزة للتطبيق:**
- **محلّي أولاً (Local-first):** كل البيانات مخزّنة على الجهاز عبر SQLite، بلا حسابات ولا سحابة ولا تتبّع.
- **أحضِر مفتاحك الخاص (BYOK – Bring Your Own Key):** المستخدم يُدخل مفاتيح API الخاصة به لمزوّدي الذكاء الاصطناعي، وتُخزَّن بشكل آمن.
- **متعدّد اللغات وRTL:** دعم الإنجليزية والعربية (مع اتجاه RTL كامل) والفرنسية.

## 2. ميزات الإصدار الأول (v1) بالتفصيل

### 2.1 إدارة البرومبتات (الجوهر)
- إنشاء/تعديل/حذف برومبت (شاشات `CreatePromptScreen` و`PromptDetailScreen`).
- حقول كل برومبت: العنوان، المحتوى، الوصف، التصنيف، المنصّة، الوسوم، المتغيّرات.
- **التثبيت (Pin)** و**المفضّلة (Favorite)** لكل برومبت، مع ترتيب يضع المثبّت أولاً ثم الأحدث تعديلاً (`ORDER BY is_pinned DESC, updated_at DESC`).
- **شاشة المفضّلة** المستقلّة (`FavoritesScreen`).

### 2.2 محرّك المتغيّرات (Variable Engine)
- ثلاث صيغ مدعومة (`variableParser.ts`):
  | الصيغة | المعنى |
  |---|---|
  | `{{var}}` | حقل نصّي حرّ |
  | `{{var\|default}}` | حقل نصّي بقيمة افتراضية |
  | `{{var:opt1\|opt2\|opt3}}` | قائمة منسدلة للاختيار |
- استخراج المتغيّرات تلقائياً من النص، وبناء البرومبت النهائي بدمج القيم (`buildFinalPrompt`).
- **مكوّن `VariableFiller`**: نموذج تفاعلي لملء القيم قبل النسخ/الاستخدام، مع حفظ **القيم الأخيرة المُستخدمة** (`recentValues`) لكل متغيّر.

### 2.3 التصنيفات والمنصّات
- تصنيفات مدمجة: صورة، فيديو، كود، كتابة، تسويق، أعمال، موسيقى، تعليم، أخرى.
- منصّات مدمجة: ChatGPT, Claude, Gemini, Midjourney, DALL-E, Stable Diffusion, Sora, Runway, Kling, Copilot, Cursor, v0, Lovable.
- **تصنيفات ومنصّات مخصّصة** يضيفها المستخدم (شاشتا `ManageCategoriesScreen` و`ManagePlatformsScreen`)، تُخزَّن في الإعدادات.
- مكوّنات داعمة: `CategoryFilter`, `PlatformBadge`.

### 2.4 البحث والتصفية
- بحث نصّي كامل عبر **FTS5** مع مشغّلات (triggers) تُبقي فهرس البحث متزامناً مع جدول البرومبتات (insert/update/delete).
- آلية احتياطية: في حال عدم توفّر FTS5، يُستخدم بحث `LIKE` على العنوان/المحتوى/الوصف/الوسوم.
- تصفية حسب التصنيف والمنصّة والمفضّلة (`PromptFilters`).

### 2.5 مساعد الذكاء الاصطناعي (`AIAssistantScreen` + `aiService.ts`)
أربعة أوضاع (Tabs):
1. **محادثة (Chat):** محادثة حرّة مع المزوّد النشط.
2. **استيراد ذكي (Smart Import):** يأخذ نصاً خاماً ويُرجِع JSON منظّماً (عنوان، محتوى مع متغيّرات، وصف، تصنيف، منصّة، وسوم، متغيّرات).
3. **توليد (Generate):** يولّد برومبتاً عالي الجودة من وصف المستخدم.
4. **تحليل (Analyze):** يُرجِع درجة جودة (1-10) واقتراحات تحسين وتحليل موجز.
- دالة `improvePrompt` لتحسين برومبت قائم، و`editPromptWithAI` لتعديله بتعليمات.
- **محلّل JSON متين** (`parseAIJson`) يتعامل مع ردود النموذج: محاولة مباشرة، ثم إزالة أسوار الـ markdown، ثم استخراج أول كائن JSON متوازن عبر عدّ الأقواس.
- تعليمات صارمة لصيغة المتغيّرات تُحقَن في كل برومبت نظام (`VARIABLE_INSTRUCTIONS`).

### 2.6 مزوّدو الذكاء الاصطناعي (BYOK)
- ثلاثة مزوّدين افتراضيين: **Google Gemini**, **OpenRouter**, **OpenAI**.
- بنية مرنة: Gemini له مسار خاص (`callGemini`)، والبقية عبر واجهة متوافقة مع OpenAI (`callOpenAICompatible`).
- شاشة `AISettingsScreen`: إدارة المزوّدين، إدخال المفاتيح، اختيار النموذج، تفعيل المزوّد النشط، واختبار الاتصال (`testAIConnection`).
- المفاتيح تُخزَّن عبر **`expo-secure-store`** (تشفير على مستوى النظام).

### 2.7 التصدير والاستيراد والمشاركة (`importExport.ts`)
- تصدير المكتبة كاملة كنسخة احتياطية (`backup`) أو برومبت واحد (`prompt`) بصيغة `.vibe` (JSON).
- مشاركة عبر `expo-sharing`.
- استيراد عبر `expo-document-picker` مع دعم صيغ JSON/نص/markdown.
- **فتح ملفات `.vibe` من خارج التطبيق** عبر Deep Links / Intent Filters (مُعرَّفة في `app.json` لأندرويد بمخططات `file://` و`content://`)، مع معالجة في `App.tsx` (`handleOpenFileUrl`) لحالتي فتح التطبيق وهو مغلق أو يعمل.
- وضعا الاستيراد: **دمج (merge)** أو **استبدال (replace)**، مع `INSERT OR IGNORE` لتفادي التكرار.

### 2.8 الواجهة والتجربة (UI/UX)
- **التنقّل:** نظام تنقّل مخصّص خفيف عبر `zustand` (`navigationStore`) بدل مكتبة تنقّل ثقيلة، مع `Navigator` و`BottomTabBar` وزر عائم `FAB`.
- زر الرجوع المادي لأندرويد مُعالَج يدوياً.
- **السمات (Themes):** خمس سمات لونية (`light`, `ocean`, `forest`, `sunset`, `dark`) عبر `THEME_PRESETS` و`useTheme`. (README يذكر "10 سمات" — تبايُن يستحق التوحيد.)
- دعم المنطقة الآمنة (Safe Area) ومعالجة لوحة المفاتيح خاصة في المحادثة (Animated keyboard avoidance).
- نافذة مساعدة (Help Modal) تشرح المتغيّرات والذكاء الاصطناعي والإعداد والأمان.
- عدّاد التوكنز التقديري (`tokenCounter.ts`).

### 2.9 التدويل (i18n)
- ملف `strings.ts` مع دالة `t(key, language)`، ودعم RTL مربوط باللغة العربية.

### 2.10 البيانات الأوّلية (Seed)
- برومبتات جاهزة عند أول تشغيل (`seed.ts`) مثل "Cinematic Portrait" و"Code Review Expert" لتعريف المستخدم بالميزات فوراً.

## 3. البنية التقنية (Architecture)

### 3.1 حزمة التقنيات
| المكوّن | الإصدار/التقنية |
|---|---|
| إطار العمل | Expo SDK ~55.0.5 |
| React Native | 0.83.2 |
| React | 19.2.0 |
| اللغة | TypeScript ~5.9.2 |
| إدارة الحالة | Zustand 5.0 |
| قاعدة البيانات | expo-sqlite (SQLite + FTS5) |
| التخزين الآمن | expo-secure-store |
| الملفات/المشاركة | expo-file-system, expo-sharing, expo-document-picker, expo-clipboard |
| أخرى | expo-haptics, react-native-gesture-handler, react-native-safe-area-context |

### 3.2 تنظيم الكود
بنية طبقية واضحة تحت `src/`:
```
src/
├── components/   مكوّنات واجهة قابلة لإعادة الاستخدام (PromptCard, VariableFiller, FAB ...)
├── constants/    التصنيفات، المنصّات، السمات، الثوابت (RADIUS, SPACING, FONT_SIZE, SHADOW)
├── database/     connection, schema (FTS5 + migrations), queries (CRUD), seed
├── engine/       منطق الأعمال: aiService, variableParser, importExport
├── hooks/        useTheme
├── i18n/         strings
├── screens/      الشاشات التسع
├── stores/       حالة Zustand: prompt, settings, navigation, history
├── types/        تعريفات TypeScript المركزية
└── utils/        clipboard, id, tokenCounter
```

### 3.3 نموذج البيانات
- جدول `prompts` بحقول: `id, title, content, description, category, platform, tags(JSON), folder_id, variables(JSON), is_favorite, is_pinned, usage_count, last_used_at, created_at, updated_at, audio_base64`.
- ملاحظات بنيوية مهمّة:
  - يوجد عمود **`folder_id`** و**`audio_base64`** في المخطط، **لكنهما غير مستخدمين فعلياً** في الواجهة — أي أنّ التطبيق مهيّأ بنيوياً لميزتي **المجلّدات** و**البرومبت الصوتي** لكنهما لم تُفعَّلا بعد.
  - **هجرة (migration)** يدوية بسيطة عبر `ALTER TABLE ... ADD COLUMN` ضمن `try/catch`.
  - الوصول للقاعدة **متزامن (Sync API)**: `getAllSync`, `runSync`, `prepareSync` — مناسب لأحجام صغيرة لكنه قد يحجب الواجهة عند الأحجام الكبيرة.

### 3.4 تدفّق التهيئة (App.tsx)
1. فتح القاعدة → تهيئة المخطط → تهيئة جدول السجلّ (history) → زرع البيانات.
2. تحميل الإعدادات من `secure-store`.
3. معالجة رابط فتح أوّلي (ملف `.vibe`).
4. الاستماع لأحداث الروابط أثناء التشغيل.

## 4. نقاط القوة (Strengths)

1. **الخصوصية والملكية:** محلّي بالكامل، بلا حسابات ولا تتبّع — ميزة تسويقية وأخلاقية قوية في زمن قلق الخصوصية.
2. **محرّك المتغيّرات:** الميزة الأبرز والأكثر تميّزاً؛ تحويل البرومبت إلى نموذج تفاعلي مع حفظ القيم الأخيرة تجربة عملية ممتازة.
3. **مرونة مزوّدي الذكاء الاصطناعي (BYOK):** بنية تجريدية نظيفة تدعم Gemini والواجهات المتوافقة مع OpenAI، فتُضاف مزوّدات جديدة بسهولة.
4. **بحث نصّي كامل احترافي (FTS5)** مع مزامنة تلقائية عبر triggers وآلية احتياطية ذكية.
5. **معالجة متينة لردود الذكاء الاصطناعي:** `parseAIJson` بعدّ الأقواس يتعامل مع فوضى مخرجات النماذج بشكل عملي.
6. **تكامل نظام التشغيل:** فتح ملفات `.vibe` مباشرةً من مدير الملفات عبر Intent Filters تجربة سلسة للمشاركة.
7. **بنية كود نظيفة ومنظّمة:** فصل واضح بين الطبقات (engine/database/stores/screens)، وأسماء aliases (`@/`) مريحة.
8. **خفّة الوزن:** تنقّل مخصّص بدل React Navigation، واعتماد قليل على المكتبات الخارجية.
9. **التدويل وRTL:** دعم عربي حقيقي من البداية — ميزة نادرة في تطبيقات مشابهة.

## 5. نقاط الضعف والمخاطر (Weaknesses & Risks)

### 5.1 تقنية
1. **الوصول المتزامن لقاعدة البيانات:** كل عمليات SQLite متزامنة (`*Sync`)؛ قد تتجمّد الواجهة مع آلاف البرومبتات أو عمليات الاستيراد الكبيرة.
2. **لا يوجد ترقيم صفحات (Pagination):** `getAllPrompts` يجلب كل الصفوف دفعة واحدة.
3. **البحث لا يستفيد فعلياً من FTS5 في الاستعلام:** `getAllPrompts` يستخدم `LIKE` دائماً رغم وجود جدول `prompts_fts` المُهيّأ — أي أنّ بنية البحث المتقدّمة مبنية لكن غير موصولة بمسار البحث الرئيسي.
4. **أعمدة ميتة:** `folder_id` و`audio_base64` موجودة بلا منطق واجهة.
5. **لا اختبارات (Tests):** لا توجد أي بنية اختبار (unit/integration) في المشروع.
6. **تكرار الإعدادات والمنطق:** أنماط persist متكرّرة في `settingsStore`؛ قابلة للتجريد.
7. **ملفات/فوضى في الجذر:** وجود `proomy-note (1).zip`، `package_last.json`، `app.json.new.json`، و`app.js` (10KB) — بقايا يجب تنظيفها.
8. **عدم تطابق التوثيق:** README يذكر 10 سمات بينما الكود يعرّف 5؛ ويذكر منصّات/لغات بصيغ مختلفة قليلاً.
9. **التخزين عبر secure-store للإعدادات الكبيرة:** `secure-store` له حدود حجم على بعض المنصّات؛ تخزين كل التصنيفات/المنصّات المخصّصة فيه قد يصطدم بحدود مستقبلاً.

### 5.2 وظيفية / تجربة المستخدم
10. **لا تنظيم هرمي:** لا مجلّدات ولا مجموعات (collections) — مع نمو المكتبة يصبح التصفّح صعباً.
11. **لا سجل إصدارات للبرومبت (Versioning):** التعديل يستبدل المحتوى بلا تاريخ تغييرات.
12. **لا نسخ احتياطي سحابي ولا مزامنة بين الأجهزة** (مقصود فلسفياً، لكنه قيد عملي).
13. **لا تحليلات استخدام مرئية:** `usage_count` و`last_used_at` يُجمَعان لكن لا توجد لوحة رؤى.
14. **لا وضع غير متصل واضح لميزات الذكاء الاصطناعي:** كلها تعتمد الشبكة بلا تخزين مؤقت للردود.

## 6. مقترحات ميزات الإصدار الثاني (v2)

### 6.1 الذكاء الاصطناعي (AI)
- **تفعيل المتغيّرات بالذكاء الاصطناعي تلقائياً:** اكتشاف الأجزاء القابلة للتخصيص واقتراح متغيّرات أثناء الكتابة (live).
- **سلاسل البرومبتات (Prompt Chaining):** ربط عدّة برومبتات في تدفّق عمل (output أحدها يغذّي التالي).
- **مقارنة النماذج (A/B):** تشغيل نفس البرومبت على عدّة مزوّدين وعرض النتائج جنباً إلى جنب.
- **تحسين تكراري موجّه:** حلقة "حسّن ← قيّم ← أعد" مع إظهار درجة الجودة قبل/بعد.
- **مزوّدون إضافيون:** دعم Anthropic Claude الأصلي، و Ollama/LM Studio (نماذج محلّية على الجهاز/الشبكة المحلّية) لتعزيز فلسفة الخصوصية.
- **تخزين مؤقت لردود الذكاء الاصطناعي** وتتبّع التكلفة/التوكنز الفعلية لكل مزوّد.
- **قوالب نظام (System Prompts) قابلة لإعادة الاستخدام** كطبقة فوق البرومبتات.

### 6.2 الإنتاجية (Productivity)
- **المجلّدات والمجموعات:** تفعيل `folder_id` الموجود أصلاً + مجموعات (collections) قابلة للمشاركة.
- **سجل الإصدارات (Version History):** تتبّع تغييرات كل برومبت مع إمكانية الرجوع.
- **القوالب السريعة والمتغيّرات العامة (Global Variables):** قيم مشتركة (مثل اسم العلامة التجارية) تُملأ تلقائياً عبر كل البرومبتات.
- **الإجراءات المجمّعة (Bulk Actions):** تحديد متعدّد للحذف/التصدير/تغيير التصنيف.
- **لوحة رؤى الاستخدام:** الأكثر استخداماً، اتجاهات زمنية، برومبتات مهملة، اقتراحات أرشفة.
- **اختصارات وودجت (Widgets) للوصول السريع** لأكثر البرومبتات استخداماً + Quick Settings tile.
- **البرومبت الصوتي:** تفعيل `audio_base64` الموجود — إدخال صوتي (speech-to-prompt) وإرفاق ملاحظات صوتية.

### 6.3 التعاون (Collaboration)
- **مزامنة اختيارية مشفّرة طرف-لطرف (E2EE)** عبر مزوّد يختاره المستخدم (مثل تخزينه الخاص) دون كسر فلسفة "محلّي أولاً".
- **مكتبة برومبتات مجتمعية:** سوق/معرض لمشاركة واكتشاف البرومبتات (`.vibe` كصيغة تبادل قائمة فعلاً).
- **روابط مشاركة عميقة (Deep Share Links)** لبرومبت واحد تفتح مباشرة في التطبيق.
- **مساحات عمل للفرق (Team Workspaces)** كميزة مدفوعة لاحقة.

### 6.4 الواجهة والتجربة (UI/UX)
- **توحيد السمات وزيادتها فعلياً إلى 10** كما يَعِد README، مع منشئ سمات مخصّص.
- **معاينة حيّة للبرومبت** أثناء ملء المتغيّرات (split view).
- **تمييز صياغي (Syntax Highlighting)** للمتغيّرات داخل المحرّر.
- **سحب وإفلات** لإعادة الترتيب وللتنظيم في المجلّدات.
- **بحث متقدّم** بمشغّلات (حسب المنصّة/التصنيف/التاريخ/عدد الاستخدام) موصول فعلياً بـ FTS5.
- **تحسين إمكانية الوصول (a11y):** قارئ شاشة، أحجام خطوط ديناميكية.
- **لمسات حسّية (Haptics)** أوسع (المكتبة موجودة `expo-haptics`).

### 6.5 التخزين والأداء (Storage & Performance)
- **نقل عمليات القاعدة إلى Async API** (`getAllAsync`/`runAsync`) لمنع تجميد الواجهة.
- **ترقيم صفحات + قوائم افتراضية (FlatList virtualization)** للمكتبات الكبيرة.
- **توصيل البحث بـ FTS5 فعلياً** مع ترتيب حسب الصلة (BM25).
- **نسخ احتياطي تلقائي مجدول** محلّياً + تصدير مشفّر اختياري.
- **استيراد/تصدير متعدّد الصيغ:** Markdown, CSV, وملفات OpenAI/Anthropic.
- **هجرات قاعدة بيانات منظّمة (versioned migrations)** بدل `try/catch` المتفرّقة.

### 6.6 ميزات مبتكرة (Innovative)
- **منشئ نماذج من برومبت (Prompt-as-App):** تحويل أي برومبت متغيّرات إلى "تطبيق مصغّر" بنموذج كامل قابل للمشاركة كرابط.
- **التشغيل المباشر (Run-in-app):** تنفيذ البرومبت داخل التطبيق عبر المزوّد النشط مباشرةً من بطاقة البرومبت، مع حفظ المخرجات.
- **اقتراحات سياقية ذكية:** "برومبتات قد تحتاجها الآن" بناءً على الوقت/النمط/المشروع الحالي.
- **سجل المخرجات (Output History):** ربط كل تشغيل بمخرجاته للرجوع والمقارنة.
- **توسيم تلقائي (Auto-tagging) وكشف التكرار/التشابه** بين البرومبتات.
- **وضع "المعمل" (Playground)** لتجربة المتغيّرات والنماذج بسرعة قبل الحفظ.
- **تكامل المشاركة على مستوى النظام (Share Sheet target):** "أرسل أي نص إلى Vibe Note" لتحويله فوراً إلى برومبت منظّم.

## 8. ملخّص تنفيذي (Executive Summary)

**Vibe Note** منتج مبني على فكرة واضحة ومتمايزة (إدارة برومبتات محلّية أولاً مع محرّك متغيّرات قوي و BYOK)، وبنية كود نظيفة ومنظّمة. أبرز أصوله: **محرّك المتغيّرات**، **تجريد مزوّدي الذكاء الاصطناعي**، و**الخصوصية الكاملة**. أبرز فجواته الحالية: **غياب التنظيم الهرمي (مجلّدات)**، **عدم توصيل FTS5 بالبحث الفعلي**، **الوصول المتزامن للقاعدة**، و**غياب الاختبارات**.

اللافت أنّ المخطط يحتوي بالفعل على بذور ميزتين كبيرتين غير مفعّلتين (`folder_id` للمجلّدات، و`audio_base64` للصوت) — ما يجعلهما **أسرع المكاسب** في الإصدار الثاني. الطريق الأمثل لـ v2 هو: **تثبيت الأساس وتنظيفه أولاً**، ثم **التنظيم والإنتاجية**، فـ **قفزة الذكاء الاصطناعي (التشغيل داخل التطبيق + المخرجات)**، وصولاً إلى **التعاون والنظام البيئي المجتمعي** الذي يحوّل التطبيق من أداة فردية إلى منصّة.

## 9. خطة الطريق التنفيذية لـ v2 (Execution Roadmap) — قابلة للتطبيق

> هذا القسم يحوّل التحليل أعلاه إلى **خطة عمل تنفيذية** مرتّبة: المنهجية، المتطلّبات المسبقة (Prerequisites)، ثم **سجلّ طلبيات الترقية (Upgrade Backlog) من v1 إلى v2** مرتّباً حسب الأولوية والاعتمادية. يُقصد به أن يكون "مصدر الحقيقة" أثناء التنفيذ، ويُحدَّث مع كل إنجاز.

### 9.1 المنهجية (Method)

نعتمد منهجية **تزايدية محكومة بالمخاطر (incremental, risk-driven)**:

1. **تثبيت البنية التحتية أولاً (Foundation-first):** قبل أي ميزة، نرفع منصّة العمل (Expo SDK، RN، التبعيات) ونزيل الديون التقنية الحرجة (FTS5، Async، الاختبارات) حتى لا نبني فوق أساس متحرّك.
2. **شرائح رأسية (Vertical slices):** كل ميزة تُسلَّم كاملة عبر الطبقات (DB → engine → store → screen → i18n) بدل تسليم طبقة كاملة عبر كل الميزات.
3. **أعلام الميزات (Feature flags):** الميزات الكبيرة (مجلّدات، تشغيل داخل التطبيق، مزامنة) خلف أعلام في `settingsStore` لتسليم آمن وتراجع سريع.
4. **هجرات قاعدة بيانات مُرقَّمة (versioned migrations):** استبدال `try/catch + ALTER TABLE` المتفرّقة بجدول `schema_version` ودوال هجرة تصاعدية مُختبَرة.
5. **بوّابة جودة لكل دفعة (Definition of Done):** يمرّ `tsc --noEmit` + `expo-doctor` + اختبارات الوحدة الحرجة + مراجعة RTL/A11y قبل الدمج.
6. **الحفاظ على الفلسفة:** كل ميزة شبكية/سحابية تبقى **اختيارية و BYOK ومحلّية-أولاً افتراضياً**؛ لا تتبّع ولا حسابات إجبارية.

### 9.2 المتطلّبات المسبقة (Prerequisites)

**أدوات وبيئة:**
- Node.js ≥ 20 LTS، npm ≥ 10، Watchman (اختياري لـ macOS).
- **Expo SDK 56** (هذه الترقية) + EAS CLI (`eas-cli`) لبناء أندرويد.
- Android Studio + Android SDK (Platform 35+) + JDK 17 لبناء `eas build -p android` أو التشغيل المحلّي.
- حساب Expo (موجود: `projectId` في `app.json`).

**ديون تقنية يجب إغلاقها قبل ميزات v2 (حاصدات سريعة):**
| # | المتطلّب | السبب |
|---|---|---|
| P0-1 | ترقية Expo 55 → 56 (RN 0.85، React 19.2.3) | منصّة حديثة + New Architecture افتراضياً |
| P0-2 | تنظيف جذر المشروع (`*.zip`, `package_last.json`, `app.json.new.json`, `app.js`) | تقليل الالتباس والوزن |

### 9.3 سجلّ طلبيات الترقية v1 → v2 (Upgrade Backlog مرتّب)

الترتيب = أولوية تنفيذ موصى بها. كل بند: **المعرّف · العنوان · الطبقات المتأثّرة · معيار القبول**.

### 9.4 واجهات/تحسينات UX مشمولة في هذه الجولة (Quick wins منفّذة)
